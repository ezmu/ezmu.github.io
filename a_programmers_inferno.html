Circle One

Architectural Collapse

Why Your Server Dies When Three Users Show Up

Let's run a simple experiment.

Open a brand new Laravel project. Or Rails. Or Spring. Or Django. Write nothing but a single line in a single route:

return "Hello";

Now start the server. Open your task manager, or run top in the terminal. Look at how much RAM this "empty" project has devoured.

What do you see? Six megabytes? Seven? Ten? In some "lightweight" frameworks, you've already passed thirty. In others, fifty or even a hundred megabytes — before you've said your first "Hello."

Thirty megabytes of RAM. Hundreds of thousands of lines of machine code. Enough to run an entire operating system from the 1980s. All so the server can say one word: "Hello."

This is the equivalent of building a full-scale water desalination plant — million-dollar pipes, industrial pumps, diesel generators, a crew of twenty — just to offer a visitor a single sip of water.

And when two visitors arrive at the same time? The plant collapses. The pumps were never designed for such "massive traffic" — two whole people.

Autopsy of a Slow Death

Let's lift the hood and see what actually happens when a user sends a request to a "modern framework" server.

Crime One: Bootstrapping Madness

When the server starts (or on the first request, depending on the framework), it performs a "warm-up" ritual worthy of launching a spaceship to Mars — except the destination is the kitchen next door.

Here's what it does:

  • Parses thousands of configuration files. Every config file, every environment variable, every default setting — read from disk, parsed, and crammed into memory. Even the settings you'll never use. Even the configurations for libraries you've never heard of.
  • Registers hundreds of "services." Modern frameworks live on dependency injection. So they register everything: database services, session services, email services, security services, caching services, queue services, translation services, validation services… all in a giant in-memory dictionary.
  • Wires everything to everything. A massive dependency tree is constructed. The database service needs a driver, a connection manager, an event listener, a query cache… all instantiated and frozen in memory.
  • Loads libraries for eternity. Libraries for JSON, XML, YAML, CSV. Libraries for making HTTP requests. Libraries for generating thumbnails. Libraries for encrypting passwords. Libraries for handling dates. Libraries for solving your marital problems. All loaded into memory from second zero.

And after all this madness… what does the server actually do? It fetches one row from a database and returns it as JSON. Or maybe just returns "Hello."

Crime Two: Blocking I/O – When Time Freezes

The moment your framework needs to do anything outside the CPU — read a file, query a database, call an external API, ask Redis for something — what happens?

It stops. It freezes. It sleeps.

This is Blocking I/O. The thread handling the user's request says: "I'll wait for the database to come back. And while I wait, I'll do nothing. I can't handle any other requests." So it sleeps. Lazy. Useless.

And while this thread sleeps, ninety-nine other threads can sleep alongside it. The entire server becomes a waiting room of napping threads, while the CPU you paid $500 for sits completely idle. Not working. Not serving. Just waiting.

The hilarious, cry-worthy paradox: today's CPUs are millions of times faster than 1980s CPUs. But because we sleep threads all the time, our servers behave as if they're forty years old.

The solution known since 2010: Non-blocking I/O. Send the request, tell the database "let me know when you're done," and go serve other users in the meantime. This exists in Node.js (from day one), in Nginx, in many modern servers. But most "production" and "easy" frameworks ignore it because it's "too complex."

The Numbers Don't Lie

An average server with a 4-core CPU and 8GB of RAM:

  • If you write clean code in C, Rust, Zig, or even Go with goroutines: you can easily handle 100,000 concurrent connections. The server will be happy. You'll be a hero.
  • If you write the same app in Laravel, Rails, or Django (default mode): at 200 concurrent users, the server starts gasping. At 500, it collapses. You'll see errors: "Connection timeout," "Out of memory," "Unable to spawn new threads."

And what do you do?

You don't fix the code, of course. Instead:

  • You buy two more servers.
  • You put a load balancer in front of them.
  • You watch your cloud bill grow like a malignant tumour each month.
  • Then you proudly declare that you've "built a horizontally scalable system."

No. You've built lazy code and discovered that laziness can be bought and sold.

A Brief History: How We Were vs. What We Became

In 1999, an Apache server with CGI could run a site like Slashdot (millions of daily views) on a single server with 256MB of RAM and a Pentium II.

In 2004, a Lighttpd server (very, very light) could serve millions of users on hardware weaker than your current smartphone.

In 2024, you need an AWS server with 8 cores and 32GB of RAM to run a simple To-do list app.

Did programming get better? No. It got much worse. But hardware got so fast that we barely notice the catastrophic decline in code quality… until the third user shows up.

When Does Your Server Really Crash?

Don't think I'm exaggerating. Ask any DevOps engineer who has worked on a Rails or Laravel app in production. I'll summarise their experience:

  • Phase 1: "I'll use this framework. It's productive and easy."
  • Phase 2: "The server hangs when users hit 100."
  • Phase 3: "I'll add Sidekiq (or Horizon) for background processing."
  • Phase 4: "I'll add Redis for caching."
  • Phase 5: "I'll add two more servers."
  • Phase 6: "My AWS bill is $5000 a month."
  • Phase 7: "Why is my app still slow?"

The answer: from the very beginning, you built a desalination plant to serve a sip of water. And now you're surprised the plant consumes energy and space.

🔥 The Bitter Verdict If your framework needs 50MB of RAM to say "Hello," then your framework isn't a tool. It's a parasitic virus. It feeds on your engineering comfort, on your server bill, on your users' patience.

"But it's a production framework!" No. A truly productive tool solves the problem with the least waste. Your framework solves it while burning a forest along the way.

"But it speeds up development!" It speeds up development until you hit Phase 6 above. Then you spend weeks debugging performance issues that could have been avoided from the start.

The Way Out (I Won't Leave You Empty-Handed)

I'm not asking you to go back to C and manual memory management. But I am asking for three things:

  • Learn how the machine actually works. Read about Stack vs. Heap. Read about Non-blocking I/O. Read about the cost of a system call. Just understand what's happening under your feet.
  • Choose your tools consciously. Do you really need a full framework? Maybe you just need a lightweight HTTP server like Express (Node.js), Fiber (Go), or Actix (Rust). The right choice eliminates 95% of the bloat.
  • Write "Hello" yourself. Try it once: use net/http in Go, http.server in Python, or std::net in Rust. Write "Hello World" without a framework. Then look at the RAM usage and response time. You'll die laughing when you see the difference between this and your bloated framework.

Last Word on Circle One

I'm not against frameworks. I'm against blindly using them for every project, as if they're the only solution.

When you build a bridge, you don't bring two excavators, three cranes, and a sand barge if all you need is to cross a small stream. You bring a wooden plank. Or you jump.

Real engineering is choosing the right tool for the problem. Not the biggest tool in the workshop.

Just because the CPU is fast doesn't mean you're allowed to be lazy. Just because RAM is cheap doesn't mean you should waste it like water.

The dignity of our profession demands that you understand what your code actually does. Not just type npm install and pretend you've accomplished something.

Circle Two

Dependency Hell

Why Importing is-even Requires 17 Packages and a Known Vulnerability

Open the Coffin

Do you know that feeling? You've been working on a project for six months. One day, you open your node_modules folder (or your language's equivalent dependency folder). You look at its size.

Go ahead. Do it now. Open any "average" project you have. Navigate to the dependencies folder. Look at the size. Not the file count — the size.

What do you see?

300 megabytes? 700? More than a gigabyte? Glory be. You have a dependency folder larger than the entire Linux operating system. Larger than most full distributions. This single folder consumes more space than a Windows 95 installation multiplied by a hundred.

Now ask yourself one simple question:

Do you understand — even approximately — what every line of code in these packages does?

Of course not.

No one does. Not you, not your colleagues, not even the authors of some of these packages. Because reaching the bottom of this tree requires dedicating yourself to reading hundreds of thousands of lines of code you've never seen, which depends on other code you've never seen, which depends on…

The Tale of is-positive (A True Story You Won't Believe)

Let me tell you about a real case from the npm archive of shame (the same story repeats on PyPI, RubyGems, Cargo, everywhere).

Someone wrote a package called is-positive. What does it do? Exactly what the name suggests: it checks if a number is greater than zero.

The actual lines of code to solve this "complex engineering problem": 5 lines. Could have been 3.

What's hilarious about this?

This 5-line package depends on 4 other packages. Each of those depends on 2–3 more. The depth of the tree reaches 7 levels in some versions.

What do those other packages do? Pretty much the same thing:

  • is-number (checks if input is numeric) depends on is-nan (checks if Not a Number) depends on type-checker depends on is-object… and so on, until you reach a package that prints the letter 'a' to the console.

This isn't code. This is a digital food chain, each link feeding on the next, and the final product is a function you could write in one line with ease.

What's even more painful?

📦 is-positive is downloaded over a million times a week. Used in thousands of projects. Your own project, right now, somewhere in its dependency tree, might be using it to check whether an ID from the database is greater than zero.
And you have no idea.

Why This Is Dangerous

1. The Security Delusion

In 2021, a serious vulnerability was discovered in the coa package (a popular package for parsing command-line arguments). When the vulnerability was announced, how many different versions of coa existed in a typical node_modules folder?

In one project I analysed: 37 different versions. Thirty-seven! Some ancient, some updated, some in paths untouched for years.

The question: how do you patch a vulnerability when you don't even know which version of which package is used where in your project?

You can't. Not easily. You'll spend days updating dependencies, resolving conflicts, testing that nothing broke. And in the end, one stale version may remain stuck somewhere. Because the current dependency system isn't designed for security. It's designed for "ease of addition."

Every package you import is a loaded gun aimed at your own foot. The more packages, the larger the attack surface. As one satirist said: "The best security package of all is deleting node_modules entirely." He's not entirely wrong.

2. Architectural Brittleness

Suppose you depend on package X, which depends on package Y. One day, the maintainer of Y decides to change the behaviour of one function (even slightly). Package X hasn't kept up with this change. What happens to your project?

Best case: it crashes at runtime. You can fix it after hours of debugging.

Worst case: it doesn't crash. It keeps running, but it does the wrong thing. Instead of checking that a number is greater than zero, it now checks that it's not zero. So it allows negative numbers where it shouldn't. Data gets corrupted. Money gets lost. You get fired.

All because package Y changed gt to gte in a minor version, and nobody noticed.

3. Intellectual Sterility

This is the worst one, by the way.

The modern programmer no longer knows how to write code. They know how to assemble code.

Watch what a "productive" developer does today:

  • Starts with npx create-blazing-app (generates a project with 500MB of dependencies).
  • Types npm install axios (HTTP library, 30 dependencies).
  • Types npm install lodash (utility function library, 70 packages behind it).
  • Types npm install mongoose (database ORM, 150 packages).
  • Types npm install jsonwebtoken (authentication, 40 packages).

At the end, they haven't written anything. They've only assembled dependencies. If you ask them: "How does lodash work under the hood?" they'll look at you like you asked for the meaning of life.

This isn't software engineering. This is the definition of a Lego builder: takes pre-made pieces from others, stacks them together, then calls themselves an "engineer."

Imagine a civil engineer who doesn't know how cement is made, how bridges bear weight, or how loads are distributed — but they assemble prefabricated bridge segments and say: "This is a new bridge." That's today's developer.

It's Not Just Node.js – The Plague Is Everywhere

Maybe you use Python and laugh at Node.js developers. Let me remind you about PyPI and the site-packages folder that can hit a gigabyte for the simplest project.

Or you live in Rust-land with Cargo, proud that dependencies are "statically linked into one binary." Great. Do you know how many packages a simple actix-web project imports? Over 250.

Or you're in Go with modules, thinking you're safe because Go "only imports what's used." You're still importing massive libraries from a stranger on the internet, and you don't know what they do.

The point isn't the language. The point is the engineering behaviour.

The Root Cause: Ethical Laziness

Why do programmers do this?

Because it's easier. No one wants to write their own is-even function. They just want npm install is-even and be done.

But pushing the cost onto someone else (the user, the environment, security, the monthly bill) isn't a solution. It's negligence.

The programmer says: "I'll use is-even because it's trusted — a million downloads." Forgetting that a million downloads doesn't mean quality code. It just means a million other lazy people made the same choice together.

Using a package for a trivial task isn't skill. It's an admission of incompetence.

💀 The Bitter Verdict The package manager is the lazy programmer's best friend and the end user's worst enemy.

The "productive" modern programmer proudly announces: "My project has 1,500 dependencies!" as if that's an achievement. It is an achievement — in wasting time, space, and security.

The only completely safe package is the one you don't import.

And the only code you fully understand is the code you wrote yourself.

The Solution: Restore the Dignity of Writing Code

I'm not asking you to write everything from scratch. There are complex packages (image processing, advanced cryptography, network protocols) that you'd be foolish to rewrite. Use them shamelessly.

But I am asking for four things:

  • Stop importing packages you could write in 10 lines. is-even? x % 2 === 0 is the whole thing. capitalize? Two lines. The famous left-pad? Less than 5 lines. Write them. You'll understand the code. You'll control it. And it will never betray you.
  • Inspect what you import. Before typing npm install something, ask: does this package contain 100x more code than I need? Does it drag 50 other packages along? If yes, find an alternative or write it yourself.
  • Occasionally read the code of packages you depend on. Not all of them. But once a week, pick a random package from node_modules, open its main file, and read 50 lines. You'll be surprised what you find. Sometimes horrified.
  • Remember the saying: "Code you don't write has no bugs." Every package you add is a security invoice you'll pay later.

The test: Go to your project now. Find the smallest package you import (by function, not size). Delete it. Write a local function to replace it in five minutes. Did you notice any difference? No. Is your project more secure? Yes — a million times. Do you feel a tiny bit of pride? I hope so.

Circle Three

The Docker Absurdity

Why You Ship a Whole Elephant in a Box to Deliver a Fly

A Journey to the Past (When Things Were Simple)

In the dawn of the software age — just 15 years ago — a programmer wrote code. That code ran directly on the operating system. There were no containers. No "orchestration." There was an executable file (or a bunch of text files), maybe some shared libraries, and that was it.

And the code was light as a feather.

I remember those days. A full website with millions of visitors running on a single server with 512MB of RAM and one CPU. And most importantly: it worked. It was fast. It was stable. The programmer knew what every line of their code did.

Then came the dark age: "But it works on my machine!" became the slogan of the lazy. Code started assuming certain things about the environment: a specific Python version, certain environment variables, pre-installed system packages, specific file paths. When environments differed between development and production, the code collapsed.

So came the container to save us… in its own sick way.

What Is a Container, Really? (Beyond the Marketing)

Marketing definition: "A container is a lightweight, isolated environment for running applications consistently across different environments."

Honest engineering definition: A container is a complete miniature operating system, with every library and tool you don't need, wrapped in an isolation layer, shipped across the network like a camel, to run a tiny app that could have been launched with a single command.

Let me illustrate with numbers.

Take a real example. A simple five-page website, connected to a PostgreSQL database. The actual requirements:

  • An executable file (or interpreted code) of 15-30MB.
  • A lightweight database of 50-100MB on disk.

Total actual: less than 150MB on disk, and less than 200MB of RAM at runtime.

Now, how does a "modern DevOps engineer" use Docker to deploy this app?

They will do the following:

Container 1: "The App"

  • Start from node:18-alpine (or some base image).
  • This image contains a mini Linux (Alpine) + Node.js + npm + system tools + build toolchains + maybe Python (because some npm packages need it).
  • Size: 180MB absolute minimum. Usually more.
  • Then add their code (15MB) inside the container.
  • Result: 200MB on disk, consuming at runtime 5x more RAM than the app actually needs.

Container 2: "The Database"

  • Start from postgres:15 (official image).
  • Contains Linux + PostgreSQL + admin tools + documentation + examples.
  • Size: 350-450MB (depending on the tag).
  • Consumes at runtime twice the RAM PostgreSQL actually needs.

Container 3: "The Reverse Proxy"

  • Start from nginx:alpine.
  • Size: 50MB extra.

A docker-compose.yml file as long as your arm (50-100 lines)

To wire these three containers together, with internal networks, volumes, environment variables, health checks, and all the complexity a "simple" website requires.

🐳 Final tally:
Image size on disk: over 600MB (compare: a simple site in 2005 took 10MB).
RAM at runtime: at least double what the app actually needs.
Startup time: seconds instead of fractions of a second.
All of this to run a site that could have been launched with two terminal commands 15 years ago.

The Paradox: Containers Don't Solve the Problem — They Confirm It

This is the central point that makes real engineers rage:

Containerization is not a solution to the problem of different environments. It's an admission of engineering bankruptcy.

Let me explain.

The real problem: the code you write isn't self-contained. It depends on hidden assumptions about the environment (specific versions, certain tools, environment variables, peculiar file layouts). Instead of fixing this pathological dependency, you package the entire environment with the code.

You're saying: "My code only works in a specific environment, so I'll ship the environment along with it."

This is like saying: "My car only works on a specific road, so I'll carry the road with me in the trunk wherever I go."

Instead of writing code that works anywhere (any standard Linux server with minimal requirements), you chose to move your entire workspace with you. That's not a solution. That's an escape.

An Analogy That Exposes the Illusion

Imagine you're a cook in a restaurant. The kitchen asks you to prepare a simple dish: fried eggs.

The proper engineering way: You go to the fridge, take two eggs, fry them in a pan with oil, serve the dish. Takes 3 minutes. The pan and kitchen are shared with everyone. Result: a clean, fast dish that consumes minimal resources.

The modern way (Docker container): You ask the restaurant management to bring you an entire portable kitchen of your own. It has a mini fridge, a stove, a pan, utensils, a sink, everything. This giant container sits in a corner of the main kitchen. You enter your container, open your fridge (which you carried with you), take two eggs (from your fridge, because you don't trust the shared fridge), fry them on your container's stove, on your container's pan. In the end, you serve the same dish. But now you've occupied 3 extra square metres, consumed twice the electricity of other cooks, and the dish took 15 minutes to prepare (because your container needed to "warm up").

Then you boast about using "advanced isolation technology for cooking."

No. You're just a fool who wasted resources and bragged about doing something simple.

Why Did Containers Spread Like This?

Because there's a difference between development convenience and production efficiency.

In development, containers are great. They guarantee every team member has the same environment. They prevent the "works on my machine" problem. That's a legitimate use case.

The problem starts when you use containers in production without thinking. When containers become the only deployment method, even for the simplest apps. When you containerise everything, and your infrastructure becomes an archipelago of bloated containers, each carrying a full operating system inside.

What happens then?

  • Your cloud bill explodes (each container requests more resources than needed).
  • Startup time increases (large images slow down auto-scaling).
  • Monitoring and logging complexity skyrockets (each container has its own log system).
  • Image sizes become obscene (600MB for an app that could be 20MB).

Then come "solutions" like Kubernetes to manage the complexity that Docker created in the first place. Instead of one container, you now have 20 containers spread across 5 servers, with an orchestrator that needs a full team to run.

We created an army of tools to fight the complexity we built with our own hands.

When Are Containers Worth the Trouble?

To be fair: there are cases where containers genuinely make sense.

  • If your app requires multiple conflicting library versions that make it impossible to run on a single system.
  • If you're running third-party apps you can't modify to make them lightweight and self-contained.
  • If you're managing a genuinely complex microservices architecture where isolating each service in a container becomes logical (but in that case, you probably have bigger problems than containers).

But most projects don't fall into these categories.

Most projects are simple web apps, medium APIs, or small 5–20 page sites. These can be deployed directly on the server as a single process (plus maybe one reverse proxy), and they'll be faster, cheaper, and simpler without containers.

Why don't people do this? Because they've heard that "containers are the right way." Because tech bloggers and tool vendors convince them that any other way is "primitive." Because everyone else is doing it.

"Everyone is doing it" is not an engineering argument. It's the definition of herd mentality.

The Alternative: Return to Simplicity (Before You Lose Your Mind)

Here's what you can do instead of containerising everything:

  • Write self-contained code. Don't depend on specific, unmanageable system tool versions. Use environment variables with sensible defaults. Avoid hidden assumptions. This isn't hard. It just takes thought.
  • Use the system package manager for dependencies. apt install or yum install has worked for 30 years. You don't need to "isolate" every dependency in its own container.
  • Use simple packaging tools. Instead of Docker, you can use tar to compress your code and ship it, and ssh to deploy it. Yes, people did this for years. It worked.
  • If you need lightweight isolation, use local sandboxing. Tools like systemd-nspawn or chroot provide similar isolation to containers without dragging a full OS along.
  • Finally, if you insist on Docker, use it wisely. Don't put it on everything. Don't use huge base images (use scratch or alpine where possible). Don't run 50 containers for a small project. One or two containers are enough for most projects.
🐘 The Bitter Verdict Docker isn't a tool. Docker is an admission of guilt.

Every time you use a container to run a small app, you're saying out loud: "My code isn't self-contained, my environment is a mess, I don't trust my operating system, and I don't know how to solve dependency problems — so I'll wrap everything and ship it as an elephant."

The container doesn't improve your code. Doesn't make it faster. Doesn't reduce bugs. It only hides the mess under a warm blanket called "isolation."

A real engineer writes code that works anywhere. A lazy engineer ships their workplace with them.

One question decides your engineering fate: Do you want to be an engineer who writes elegant solutions, or a plumber who carries all their pipes everywhere they go?

Circle Four

Cloud Conspiracy & Jevons Paradox

Why "Performance Optimization" Is a Myth Sold by Those Who Profit from Your Sloth

A Moment of Revelation

Let's talk about the deepest betrayal of the technology age. Not an overt moral betrayal, but a cold, calculated, paid-for betrayal dressed in the clothes of "progress" and "modernity."

Over the past three circles, we've exposed:

  • How code is born sick (Architectural Collapse).
  • How we multiply the sickness (Dependency Hell).
  • How we wrap the sickness instead of curing it (The Docker Absurdity).

And the question that has hung in the air all this time is:

Why doesn't anyone stop this madness?

Why doesn't Google issue an internal directive banning React because it's too heavy? Why doesn't AWS force customers to reduce RAM consumption? Why doesn't Intel release "slow" cheap processors for developers who write efficient code?

The answer is simple and shocking:

💰 Digital bloat isn't a bug. It's a business model.

Beneficiary #1: Hardware Vendors (Intel, AMD, Apple)

Every two years, Intel (or AMD or Apple) announces a new processor "30% faster than the previous generation." The press conference is full of charts and numbers. Journalists pant. The crowd cheers.

Then what happens? Manufacturers (Dell, HP, Lenovo) buy these processors. They make "more powerful" devices. Consumers buy them. They throw away their old devices (which were still working perfectly). The cycle continues.

But here's the forbidden engineering question:

What if developers kept writing lean, efficient code, like they did in the 1990s?

What if a web app today consumed the same resources it did in 2005 (50MB RAM instead of 500)?

A Core 2 Duo from 2006 would still be sufficient for modern web browsing. You wouldn't need an i9, an M3, or a 12-core processor. You wouldn't need to upgrade your device every 3 years. You wouldn't buy new processors.

And thus: hardware vendors would go bankrupt.

This isn't a conspiracy theory. This is pure economics. Hardware vendors need a compelling reason for you to buy their new product. And if old code is good, lean, and efficient, you won't find that reason.

So they desperately need bloated code that creates an "artificial need" for faster processors, more RAM, faster storage, bigger batteries.

Every performance flaw in your code is a profit opportunity for a hardware vendor somewhere.

Beneficiary #2: Cloud Providers (AWS, Google Cloud, Azure)

Let me ask you an embarrassing question:

How many servers do you actually need to run your app? Not how many you're buying. How many you'd actually need if your code were ideal.

If you write your app in Go or Rust with a lightweight web server like Fiber or Actix, you can handle thousands of requests per second on a tiny server with 1 vCPU and 1GB of RAM.

But… you're using Laravel or Rails with Apache, a separate database, Redis, Sidekiq, and all the microservices you don't need. You need at least two servers (one for the app, one for the database). Probably more.

Now, how much are you paying AWS each month? Probably 5 to 10 times more than you need.

Will AWS ask you to optimise your code to reduce consumption? Of course not. They'll send you a larger bill and suggest "savings plans" that make you pay slightly less (but still more than if your code were efficient).

Cloud providers profit when your code is bad. The slower and heavier your code, the more vCPUs you need, the more RAM, the more storage, the more extra services (CDN, Load Balancer, Auto-scaling groups). Your bill grows. Their profits grow.

Do you think AWS makes $80 billion a year because developers write ideal code? No. They make it because developers write lazy code.

Beneficiary #3: Framework and Library Maintainers (The "Heavy" Ones)

Do you think the React team (from Facebook/Meta) works for free? Do you think Laravel's maintainers (Taylor Otwell and team) live on love alone?

There's money behind these projects. Companies sponsor them. And those companies have interests.

If a framework is heavy and complex:

  • It becomes hard to leave (due to investment).
  • It requires training courses, consultants, and "experts."
  • It demands more servers, more cloud, more hardware.
  • It becomes an "industry standard" you can't avoid.

Notice how most heavy frameworks come from big companies:

  • React ← Facebook/Meta (profits from ads, profits by keeping users on their heavy sites).
  • Angular ← Google (profits from ads, cloud, data).
  • .NET ← Microsoft (profits from Windows licenses and Azure servers).
  • Spring ← Pivotal/VMware (profits from enterprise support and consulting).

No big company promotes an ultra-light framework that any beginner can learn in two days. Because there's no money in that. Money is in complexity.

Jevons Paradox – The Theoretical Weapon

Let me introduce an economic concept that explains everything we're seeing.

Jevons Paradox (named after economist William Stanley Jevons, 1865):

📉 "As the efficiency of using a resource increases, total consumption of that resource increases, not decreases."

The explanation: when steam engines became more efficient (consuming less coal per unit of power), running them became cheaper. So demand increased. They were used in more factories, more trains, more ships. Ultimately, total coal consumption increased instead of decreasing.

Now apply this to software:

Every time we build a faster processor, developers invent a slower framework to consume all that speed again.

The software Jevons Paradox: hardware improvements (faster, cheaper per operation) lead to software behaviour that consumes those improvements entirely — and demands even more.

Practical application:

  • 1995: 100MHz CPU, light browser, 50KB web pages. Site works fast.
  • 2005: 2GHz CPU (20x faster), heavier browser, 500KB pages. Site works at same speed (or slightly slower).
  • 2015: 3GHz CPU with 4 cores (~120x faster than 1995), 2MB pages (40x larger). Site still… wait… still slow.
  • 2024: 5GHz CPU with 12 cores (~500x faster), 5-10MB pages (100-200x larger). Site is still slow.

Where did all that speed go? Consumed by bloat. Consumed by heavy frameworks, dependencies, uncompressed JavaScript, ads, trackers.

Software hasn't become better than it was in 2005. It's become worse. But hardware got so much better that we didn't notice the decline — until we stopped and measured.

The Psychology of Distraction

How Skeleton Loaders and the "Smooth Experience" Lie Hide Your Shame

  • Then a spinning coffee icon twirls hypnotically.
  • Finally… after 15 seconds of waiting, the waiter brings the coffee.

What would you do? You'd leave. Or scream. Or smash the place. Or write a furious Google review. Or all of the above.

In the world of web and mobile apps, this behaviour isn't horrifying. It's called "best practices." And it has beautiful names:

  • Skeleton Loaders
  • Shimmer Effects
  • Fancy Progress Bars
  • Beautiful Spinners
  • Splash Screens
  • Skeleton Screens

🎭 Our code is slow. We're ashamed of it. So instead of fixing it, we put makeup on its sluggishness.

Why Do We Need These Indicators in the First Place?

Let's be honest for a moment.

If your app displayed content within 50 milliseconds (half a tenth of a second), would you need a Skeleton Loader? No. By the time the animation appeared, the content would already be there and gone.

If your server responded within 100 milliseconds, would you need a fancy progress bar? No. The user wouldn't even notice the delay.

Therefore, all these "visual improvements" are nothing but covers for slowness. They're like putting thick makeup on an open wound. The wound is still there. You're just making its appearance less unpleasant.

The modern programmer says: "My app takes 3 seconds to load the homepage, so I'll add a cool Skeleton Loader to make the user feel like something is happening."

The correct engineering response: "Why does your app take 3 seconds? Fix that first. Then talk about the user interface."

Dissecting the Skeleton Loader (Or, How to Trick the User's Eye)

What exactly is a Skeleton Loader?

It's a ghost version (placeholder) of the final content's shape, displayed while the real content loads in the background. It looks beautiful. It gives the feeling that the app is "alive" and "responsive."

But functionally, a Skeleton Loader is an organised visual lie.

You're telling the user visually: "The content is here. Almost. Wait a bit. It's coming. Look at these pretty shapes moving."

But the truth: the content isn't here. The server is still gasping for data. The network is still moving bits. The database hasn't returned the result yet.

A Skeleton Loader is a false promise wrapped in beautiful design.

The Grand Crime: Skeleton Loader + Microservices = Slow-Motion Disaster

This is where it gets tragic.

Imagine a "modern" app built on microservices. A user profile page, for example, needs:

  • User service — to fetch name, photo, email.
  • Posts service — for the last 10 posts.
  • Likes service — for like counts on each post.
  • Comments service — for the last 3 comments.
  • Stats service — for follower counts, following counts.

Instead of calling these services intelligently (or better: merging them into a single database query from the start), the app makes 5 sequential or parallel HTTP requests.

The result:

  • Second 0–0.5: skeleton for basic info.
  • Second 0.5–1: name and photo appear (after first response arrives).
  • Second 1–1.5: skeleton for posts section.
  • Second 1.5–2.5: posts appear (without likes).
  • Second 2.5–3.5: likes and comments appear.
  • Second 3.5–4: stats appear.

The user sits staring at a screen that changes every second — a mix of real content, skeleton shapes, and spinning loaders. They feel like they're watching an endless surrealist show. After 4 seconds… the page is finally complete.

If the app were written properly (one server-side query returning everything at once), all the content would appear within 0.5–1 second. No Skeleton Loader needed at all.

The Dark Side: Motion Animations as Addiction

What do animations have to do with all this?

The modern user is addicted to motion. Whenever they see something move, they think the app is "alive" and "fast." Even if the motion hides monstrous slowness.

Look at popular mobile apps (Facebook, Twitter, Instagram). Every tap produces motion: a button grows and shrinks, a menu slides from the left, a screen transitions smoothly, loading indicators spin.

Each of these motions takes time. 200–300 milliseconds per motion. If your app has 10 motions in the user's journey, you've added 2–3 seconds of pure waiting time — unrelated to network or server — artificial waiting because the designer wanted it to be "beautiful."

Motion doesn't make the app faster. Motion makes the app slower, but it psychologically numbs the pain of waiting.

This is like giving a child a lollipop while drawing blood. The pain doesn't disappear. You're just distracting them.

Debunking the Lie: What Designers Say vs. What Actually Happens

The User (inside their head): "Why is this app so slow? I could have read three posts in the time this skeleton took to appear."

The Real Engineer (if one exists): "If we fixed the fundamentals (server speed, image compression, reduced JavaScript, merged requests), we wouldn't need any of this makeup."

But no one listens to the Real Engineer. Because the Real Engineer says uncomfortable things: "Our code is bad. Let's fix it." The Designer says comfortable things: "Let's add some nice motion." Management loves nice motion. The user pretends to be satisfied.

When Are Loading Indicators Legitimate (Just a Little)?

To be fair: there are rare cases where a real loading indicator is necessary.

  • Operations that unavoidably take over a second: uploading a large file, processing video, calling a slow external API. Here an indicator is necessary because you can't reduce the time.
  • Asynchronous operations with unknown duration: payment confirmation, waiting for a human response. Here the progress bar is fake anyway (since you don't know how long it will take), but its presence reassures the user that the process "hasn't frozen."
  • Apps with genuine constraints: a satellite connection, a remote rural area. Here some slowness isn't entirely the developer's fault.

Outside these cases, loading indicators are covers for engineering laziness.

The Solution: Expose the Truth, Don't Decorate It

What if we flipped the equation?

Instead of hiding slowness, let's show it to developers.

Imagine that Developer Tools in the browser displayed, for every page, two numbers:

Imagine these numbers were displayed on the team dashboard, and management asked: "Why is half our load time 'fake waiting'?"

I think Skeleton Loaders would disappear within a week.

Transparency is the solution. Don't decorate slowness. Expose it. Then fix it.

The Journey of a Single Data Packet (From Finger to Atmosphere)

Let's trace the path of one "Like" tap. Just a tiny click on a heart-shaped button.

Stop 1: Your Phone (The Source)

  • You tap the button. Your phone crafts a small data packet (HTTP request). Its size is just a few hundred bytes (about 0.0005MB). Tiny. No problem here.
  • But… the phone consumes electricity to send this packet (radio, CPU, RAM). Each tap consumes a tiny fraction of battery. The battery is charged from electricity. Electricity mostly comes from burning fossil fuels (in most of the world).
  • Rough estimate: every million "Like" taps consumes electricity equivalent to running one LED bulb (10 watts) for 10 days. Seems small. But wait — this is just the phone.

Stop 2: The Cell Tower and Routers (The Road)

  • The packet travels wirelessly to the nearest cell tower. The tower runs on electricity. Cell towers consume enormous power (cooling, AC, transmitters).
  • From the tower, the packet travels through fibre optic cables. These cables stretch thousands of kilometres, across ocean floors and underground. Every router along the way consumes electricity and processes the packet.
  • Estimate: sending 1GB of data across the global network consumes about 5-7 kWh of electricity (due to all the intermediate equipment). That's like running an electric oven for 3 hours.

Stop 3: The Data Centre (The Real Furnace)

  • Here's the real crime. A data centre is a cooling lab with computers inside.
  • The application server receives the "Like" request. It processes it: authenticates the user, writes a row to the database, updates the cache, sends a notification (maybe), logs the event.
  • All these operations consume CPU cycles and I/O operations.
  • But worse: the server gets hot. Very hot. Modern CPUs consume 100-300 watts per server. With hundreds of thousands of servers in one data centre, the heat becomes immense.

Stop 4: The Cooling System (Double Crime)

  • Servers can't function if temperatures exceed 25-30°C. So data centres need massive cooling.
  • The cooling system consumes electricity equal to 30-50% of the servers' own consumption.
  • Even worse: some data centres use water cooling. Yes, drinking-quality fresh water is used to cool servers, then evaporates or is discharged.
  • Estimate: an average data centre (10 megawatts) consumes daily water equal to a small town of 2,000 people.

Stop 5: The Return Trip (Same Path, Same Cost)

  • After processing the "Like," the server sends a small response (200 OK) back along the same path. Same towers, same routers, same cables. Roughly the same energy consumption.

Final Stop: The Atmosphere (Carbon Dioxide)

  • Every kWh of electricity generated from fossil fuels releases about 0.5 kg of CO₂ (the number varies by source, but this is a global average).
  • Each server consuming 300 watts (including cooling) means 0.3 kWh per hour of operation.
  • Multiply this by thousands of servers, millions of requests, millions of hours… the numbers become astronomical.

Circle Eight

Digital Colonialism & The Wall of Abstraction

Who Did You Exclude from Technology Today?

A Painful Moment of Honesty

Throughout seven circles, we've talked about bloated code, insane dependencies, oversized containers, cloud collusion, distraction psychology, carbon emissions, and 4K depravity.

We've been talking about technology, about performance, about the environment. As if these problems affect everyone equally.

But they don't.

There is a class of humans who suffer more from digital bloat than anyone else. Not because their devices are slower (though they are). Not because their internet is weaker (though it is). But because our entire technical system was built by the rich, for the rich, assuming every user has the latest iPhone and the fastest internet.

This isn't negligence. This is digital colonialism.

The Test You Dare Not Run

Let's do a simple experiment. I dare you to do it today.

Test One: Slow Network

Open your favourite website (or the app you built yourself). Now, in your browser's Developer Tools, find "Network Throttling." Select "Slow 3G" (speeds similar to what half the world's population uses, especially in developing regions or rural areas).

Now load the page. How long did it take? 10 seconds? 20? 30? Did any content appear at all, or just a skeleton loader and a spinning indicator that never stops?

On many "modern" apps, the page takes over 30 seconds on Slow 3G. Often, it never loads at all.

Test Two: Old Device

Find any Android phone from 2018 (like a Samsung Galaxy S8 or Huawei P20). Or an iPhone from the same era (iPhone 8 or X). These devices are still used by hundreds of millions of people worldwide.

Load your favourite app (or your own app) on this phone. Does it run smoothly? Or does the phone heat up like a toaster, lag on every scroll, and drain the battery in 3 hours? Most likely, the second.

🧱 The painful result: Your "modern," "beautiful," "cutting-edge" app doesn't work on the devices of 70% of the world's population. You're building exclusively for the elite (those with latest devices and fastest internet). The rest… don't matter.

Numbers That Don't Lie: The Device & Internet Gap

  • Internet speed: Global average fixed broadband is about 80Mbps. But averages hide the gap. In African countries, the average is below 10Mbps. In rural India, Brazil, or Indonesia, speeds can be 2-3Mbps.
  • Smartphones: Over 3 billion people use smartphones. But the vast majority are low-spec devices (less than 2GB RAM, 32GB storage, weak processors).
  • Computers: Over 1.5 billion computers in use worldwide. Nearly half are over 5 years old. Many run old operating systems with 4GB of RAM or less.

The bottom line: As a developer, if you only test on the latest iPhone with 5G, you're ignoring more than half of humanity. And that half isn't "marginalised" or "technologically backward." They're humans like you, wanting to use digital apps and services. But you're excluding them — accidentally or deliberately.

The Wall of Abstraction: How We Robbed Programmers of Machine Understanding

This is the other dimension of the disaster. Not only are poor users victims, but programmers themselves — especially newer ones.

What is the Wall of Abstraction? It's the successive layers of abstraction between the code the programmer writes and what actually happens in the silicon (CPU, memory, disk, network).

In the past (before 2000 approximately):

  • The programmer wrote either in C (close to the machine) or Assembly (practically machine language).
  • The programmer knew where every byte went in memory.
  • The programmer knew how many CPU cycles a function cost.
  • The programmer could read a memory map and understand it.

Today (after 2020):

  • The programmer writes in a high-level language (Python, JavaScript, Java, C#) on top of a framework on top of libraries on top of libraries.
  • The programmer doesn't know where the bytes go. Doesn't know CPU cycle costs. Doesn't even know if their code will finish in 100ms or 10 seconds.
  • The programmer thinks array.sort() is "magic" with no cost.
  • The programmer doesn't know the difference between O(n) and O(n²), and doesn't care, because "arrays are small" (then the project grows and collapses).

This is the Wall of Abstraction. It separates the programmer from the machine's reality. It makes them ignorant of the true cost of their operations.

And this isn't entirely the programmer's fault. It's the fault of the education system (teaching frameworks instead of computer science), and the industry culture (valuing "speed of development" over "understanding fundamentals").

The Result: A Generation of "Software Engineers" Who Don't Understand Software

Let me describe a few real characters (names anonymised, but stories real):

  • Ahmed: Bootcamp graduate, 6 months of training. Knows React and Node.js. Doesn't know the difference between Stack and Heap. When asked to optimise a function taking 3 seconds, he suggested adding a Skeleton Loader to "improve user experience." Never once thought to rewrite the function itself.
  • Sara: Java programmer for 5 years. Uses Spring Boot and Hibernate. Doesn't know how Java's Garbage Collector works. When her app freezes (because GC runs for 2 seconds), she says: "It's Java's fault, not my code."
  • Khalid: "Professional" web developer using Laravel. When asked: "How many requests per second can your server handle?" He stared blankly. He had never measured it. Doesn't even know how to measure it.

These aren't stupid people. They're very smart. But they're victims of the Wall of Abstraction. They learned to use tools, not to understand fundamentals. The result: bloated code, wasted hardware, frustrated users.

Wall of Abstraction × Device Gap = Digital Colonialism

Now let's connect the two.

  • The dark side for users: Their devices are old, their internet slow. The bloated code written by developers (who don't understand cost) makes these devices unable to run modern apps.
  • The dark side for developers: They don't understand the true cost of their code. They live in the illusion "hardware is fast enough" because they only test on the latest devices.
  • The final result: A growing digital divide. The rich (latest devices, fast internet, developers who understand fundamentals) enjoy a smooth digital experience. The poor (old devices, slow internet, beginner developers who never learned fundamentals) are excluded from this experience.
🌍 This isn't "natural technological progress." This is digital colonialism with a "new age" flavour. Instead of ships and guns, we use bytes and frameworks. Instead of occupying land, we monopolise the smooth digital experience for those who can afford it.

What Can You Do as a Programmer (To Break the Wall)?

This is the most important section of this entire circle.

  • Learn the fundamentals. Not all of them, but enough to understand cost. Study: Data Structures, Algorithm Complexity, Memory Management, Computer Architecture, Networking.
  • Test on real, bad devices. Don't test only on your new MacBook Pro. Use an old phone, or a virtual machine with weak specs (1 core, 1GB RAM, slow network). If your app doesn't run there, it's not a global app.
  • Read the code you use. Don't trust npm install or pip install blindly. Occasionally read the code of the libraries you depend on. You'll be surprised (and sometimes horrified).
  • Write simple code yourself. Instead of importing a date-formatting library, write it. Instead of a package to check odd/even, write it. These small exercises build your machine-understanding muscles.
  • Always ask: "How much does this line cost?" How many CPU cycles, how many bytes of memory, how many network requests, how many grams of CO₂. Make cost part of your code evaluation.
🏛️ The Painful Verdict You're building digital walls higher than the walls of material poverty.

Every extra line of code is a brick in this wall. Every package you import without need is a lock on a door. Every heavy framework you choose is a concrete barrier.

The result: the poor (with old devices and slow internet) cannot enter your beautiful digital world. They remain outside, looking in, hearing about the "digital revolution" that doesn't work on their devices.

And you, in your glass palace (latest Mac, fastest internet), don't see these people. Don't know them. Don't think about them. Because you're busy "improving user experience" (for users just like you).

This is digital colonialism: not military occupation, but technical exclusion dressed in "progress" and "modernity."

If you're a respectable programmer, you'll open your eyes now. You'll see there's a world beyond your 4K screen and fibre optic connection. You'll rethink your code. You'll test on slow devices. You'll learn fundamentals. You'll break the Wall of Abstraction.

Or… you'll keep building higher walls, and call it "development."

The choice is yours. But remember: every wall you build keeps someone outside the digital civilisation. Is that the message you want to send?

Circle Nine

The Manifesto – Return to Simplicity or Die Slowly

Our Engineering Duty to Logic, the Planet, and Dignity

Before You Begin: Contemplate This Truth

You've read eight circles. You've seen how servers collapse, how dependencies create security hazards, how containers wrap failure, how the cloud and hardware profit, how skeleton loaders hide slowness, how carbon rises from every "Like," how 4K auto-play burns fuel for no reason, and how the Wall of Abstraction separates programmer from machine and excludes the poor.

Now, after all this diagnosis, it's time for the cure.

I won't offer you magic solutions. I won't sell you a "revolutionary" new framework. I won't say "go back to Assembly or C." That's romantic nonsense with no benefit.

I will give you six principles. Not rigid laws, but a compass. If you follow them, you'll write code that respects the machine, the user, and the planet. If you ignore them, you're part of the problem, not the solution.

This is The Simplicity Revolution. Join or leave.

Principle One: Ask Before You Write

Before you add any line of code, before you import any package, before you choose any framework, ask yourself:

  • Do I really need this?
  • Can I achieve the same result with 10 lines instead of 100?
  • Is this feature essential for the user, or am I adding it because it's "beautiful" or "trendy"?

If the answer is "I don't know," don't add it. Wait until you know.

This principle is called Deliberate Simplicity. The opposite of "engineering noise" — adding layers, dependencies, and features because we can, not because we need.

Remember: The best code is the code you don't write. Every line you delete is one less security hole, one less carbon emission, one less cause of slowness.

Principle Two: Understand What's Under Your Feet

You cannot optimise what you don't understand.

If you write JavaScript, understand how the Event Loop and Garbage Collector work. If you write Python, understand the difference between list and tuple. If you write Java, understand how the JIT Compiler and Memory Model work. If you write Go, understand Goroutines and Channels. If you write anything, understand how memory is managed, how the network works, and the cost of a system call.

Ignorance of cost is not an excuse. It's professional negligence.

Principle Three: Measure Before You Brag

"My app is fast!" How do you know? Did you measure it?

If you haven't measured Load Time, Time to Interactive, memory consumption, CPU usage, data consumption, carbon emissions… then you know nothing about your app's speed. You're just assuming.

If you don't measure it, you don't improve it. Numbers don't lie. Excuses don't matter.

Principle Four: Less is More (In Everything)

This principle applies to every engineering decision:

  • Lines of code: Fewer lines = fewer bugs = less complexity = easier maintenance.
  • Number of packages: Every package you add is a potential security hole, an extra dependency, a memory burden.
  • Number of layers: Every extra abstraction layer moves you away from the machine and makes optimisation harder.
  • Number of features: A feature 90% of users don't use is a burden on the 10% who need it and on the developers who maintain it.
  • Number of services: Don't turn a simple project into 15 microservices before you reach 1000 users. A single monolith is much easier at the start.

Remember: Addition is easy, deletion is hard. Start with the minimum you need, then add when necessary.

Principle Five: Test on Real, Crappy Devices

Don't test only on the latest iPhone with 5G. Test on:

  • An Android phone from 2018 (or an emulator with weak specs).
  • A "Slow 3G" network connection (or slower).
  • An old laptop with only 4GB of RAM.
  • An old browser because some users don't update.

If your app doesn't work on these devices, you're building only for the rich. You're contributing to the digital divide. You're part of digital colonialism.

Test on the weakest device your real user might have. Then optimise until it runs smoothly. Only then can you say you're building for everyone.

Principle Six: Remember Carbon in Every Decision

Every CPU cycle, every byte in memory, every bit on the network, every request to the server, every row in the database… has a carbon cost.

It may be tiny per operation. But multiply by millions of operations, and it becomes mountains of CO₂.

Every line of code you write, every byte you send, consumes the planet's resources. Make it worth it.

The Final Statement: The Simplicity Revolution

You don't need a new framework. You don't need a revolutionary programming language. You don't need a training course on "best practices." You don't need a DevOps consultant to turn your project into 15 containers.

All you need is to stop. Think. Ask: "Is this really necessary?"

This revolution doesn't happen at conferences or on tech blogs. It happens in your text editor, in your daily decisions, in your refusal to add another package, in your insistence on understanding what your code does.

You, as a programmer, hold immense power. You decide how the digital world behaves. You decide how slow or fast it is. You decide how heavy or light it is. You decide how fair or unfair it is. You decide how clean or polluted it is.

Don't blame "users who want too many features," or "management that pushes for speed," or "the market that demands modernity." You're the engineer. You're responsible. You write the code.

⚔️ If, after reading these nine circles, you still believe your giant framework is "necessary," that containers are a "magic solution," that 4K auto-play is an "excellent experience," that thousands of dependencies are "best practices"... then you're not an engineer. You're just a tool operator.

Leave. And leave engineering to those who deserve it.

But if you feel righteous anger, and decide to open your editor tomorrow and write "Hello World" without a framework, to measure its performance, to test it on an old device, to calculate its carbon, to ask "Is this really necessary?"… then welcome to The Resistance Against Bloat.

It won't be the easiest path. But it's the only path worthy of the name "engineering."
